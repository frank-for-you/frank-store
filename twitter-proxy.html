What's planned

  • 🔐

    Auth Layer

    Handles OAuth 2.0 Bearer tokens and user-level OAuth flows. Your app talks to the proxy with a simple API key — the proxy manages the upstream Twitter credentials. No more rotating tokens in your app config.

  • ⚡

    Response Caching

    Configurable TTL-based caching for read endpoints. User timelines, profile lookups, search results — cache them for as long as makes sense for your use case. Dramatically reduces API usage and improves response times.

  • 🚦

    Rate Limit Management

    Tracks rate limit headers per endpoint and per credential. Queues requests when limits are approaching. Returns meaningful errors (with retry timing) instead of letting requests fail silently.

  • 📦

    Request Deduplication

    If two requests for the same resource arrive within the cache window, the proxy collapses them into a single upstream call. Useful when multiple users or services are fetching the same data.

  • ⚙️

    Endpoint Configuration

    Allowlist and denylist specific Twitter API endpoints. Set per-endpoint cache TTLs and rate limit strategies. All config-driven — no code changes needed to adjust behavior.

  • 📊

    Observability

    Every proxy request logged in structured JSON: upstream latency, cache hit/miss, rate limit state, and error details. Know exactly what's happening without guessing.

FAQ

Why would I need a proxy for the Twitter API?
The Twitter/X API has aggressive rate limits, inconsistent auth, and no built-in caching. If you're building anything that reads from Twitter at any scale — dashboards, monitoring tools, bots, data pipelines — you'll hit limits fast. A proxy layer centralizes the credentials, caches aggressively, and handles the rate limit headaches so your app doesn't have to.
Does this require a paid Twitter API tier?
The proxy works with whatever tier you have. It doesn't increase your limits — it just helps you use them more efficiently through caching and deduplication. Some endpoints require a paid tier; the proxy can't change that.
When will this ship?
This is currently in the planning stage — later in the roadmap than Agent Setup Kit and PRD Templates. The timeline will depend partly on demand signals. Let us know if you're interested — it helps prioritize.
What's the deployment model?
Self-hosted. You run it on your own infrastructure — a VPS, a container service, wherever you host your other backend services. It's designed to be deployed alongside your existing stack, not as a third-party SaaS you depend on.
